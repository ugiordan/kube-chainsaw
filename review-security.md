## Finding 1: Suppression file has no size limit, enabling resource exhaustion

**Severity: MEDIUM**
**File:** `pkg/suppression/suppression.go`, line 26
**Category:** Resource Exhaustion

`LoadSuppressions` calls `os.ReadFile(path)` with no file size check. Unlike the manifest loader (which enforces `MaxFileSize`), the suppression loader reads the entire file into memory regardless of size. A maliciously large or accidentally huge suppressions file would cause unbounded memory allocation.

```go
func LoadSuppressions(path string) ([]Suppression, error) {
	data, err := os.ReadFile(path)  // no size limit
```

**Recommendation:** Add a size check (e.g., `os.Stat` + compare against a reasonable max) before calling `os.ReadFile`, consistent with the pattern used in `loader.processFile`.

## Finding 2: Symlink TOCTOU race in directory walking

**Severity: MEDIUM**
**File:** `pkg/loader/loader.go`, lines 65-87 and 104-131
**Category:** Path Traversal / Race Condition

At the top level, `LoadManifests` calls `os.Lstat(p)` to check for symlinks, then later calls `walkDir(p, ...)` or `processFile(p, ...)`. Between the `Lstat` check and the actual file operations, the file system entry could be replaced with a symlink (TOCTOU race). An attacker with write access to the scanned directory could exploit this to read files outside the intended scan scope.

Inside `walkDir`, the code checks `d.Type()&fs.ModeSymlink != 0`, but `filepath.WalkDir` does not follow symlinks by default and `DirEntry.Type()` reflects the entry itself. However, `processFile` then calls `os.Stat(path)` (which follows symlinks) and `os.ReadFile(path)` (which also follows symlinks). So a symlink appearing between the `WalkDir` entry check and the `processFile` call would still be followed.

**Recommendation:** Use `os.Lstat` instead of `os.Stat` in `processFile` to avoid following symlinks. Alternatively, open the file with `os.Open` and use `Fstat` on the file descriptor to atomically check the file type and size before reading.

## Finding 3: os.Exit(1) bypasses deferred cleanup and proper error propagation

**Severity: LOW**
**File:** `cmd/kube-chainsaw/main.go`, line 131
**Category:** Error Handling / Correctness

The `run` function calls `os.Exit(1)` directly when findings exceed the severity threshold. This is inside a `cobra.Command.RunE` function, which means cobra's own error handling, any deferred functions in the call stack, and output file flushing could be skipped. If future code adds deferred resource cleanup (temp files, profiling, etc.), this will silently break.

```go
for _, f := range findings {
    if !f.Suppressed && f.Severity >= failSeverity {
        os.Exit(1)
    }
}
```

**Recommendation:** Return a sentinel error instead of calling `os.Exit` directly. Handle the exit code in `main()` after `rootCmd.Execute()` returns:
```go
if errors.Is(err, errThresholdExceeded) {
    os.Exit(1)
}
```

## Finding 4: GitHub Action downloads binary over unauthenticated channel without checksum verification

**Severity: HIGH**
**File:** `.github/action.yml`, lines 35-41
**Category:** Supply Chain

The GitHub Action downloads the binary using `curl -sL` from the GitHub releases API, then pipes it directly to `tar xz` with no integrity verification (no checksum comparison, no signature check). If the GitHub release is compromised, or if an attacker MITMs the download (unlikely with HTTPS but possible with CA compromise), arbitrary code runs in the CI runner with the runner's permissions.

The "latest" resolution path is additionally fragile: it trusts `api.github.com` JSON to provide the download URL, parses it with `grep` and `cut`, then follows the redirect.

```bash
curl -sL "${DOWNLOAD_URL}" | tar xz -C /usr/local/bin kube-chainsaw
```

**Recommendation:**
1. Download the `checksums.txt` file (generated by goreleaser) alongside the binary.
2. Verify the binary's SHA256 against the checksum file.
3. Optionally sign the checksum file with cosign/GPG and verify the signature.
4. Pin to a specific version in examples/docs rather than defaulting to `latest`.

## Finding 5: Map key collision silently overwrites earlier resources

**Severity: MEDIUM**
**File:** `pkg/loader/loader.go`, lines 208-251
**Category:** Logic Error / Data Integrity

The `categorize` function stores ClusterRoles by name and Roles by `namespace/name` in maps. If two YAML files define the same ClusterRole name (or same namespace/name for Role, ServiceAccount, Pod), the second one silently overwrites the first. In a real scan across a large repo, this means findings for the first occurrence are lost without warning.

This is especially dangerous because an attacker could place a benign-looking duplicate role definition later in the directory tree to shadow a malicious one.

```go
result.ClusterRoles[name] = &models.ClusterRoleData{...}  // silent overwrite
```

**Recommendation:** Detect duplicate keys and either:
- Emit a warning finding (e.g., "KC-016: Duplicate ClusterRole definition")
- Analyze both definitions
- At minimum, log the conflict

## Finding 6: YAML bomb / alias expansion not mitigated

**Severity: MEDIUM**
**File:** `pkg/loader/loader.go`, line 172
**Category:** Resource Exhaustion

The code uses `sigs.k8s.io/yaml` (which wraps `go-yaml`) to parse YAML documents. YAML anchors and aliases (`&anchor` / `*alias`) enable exponential expansion attacks (YAML bombs, similar to XML billion laughs). A malicious YAML file like:

```yaml
a: &a ["lol","lol","lol"]
b: &b [*a,*a,*a,*a]
c: &c [*b,*b,*b,*b]
d: &d [*c,*c,*c,*c]
```

While `go-yaml` v3 has some built-in limits on alias dereferencing depth, `sigs.k8s.io/yaml` goes through `encoding/json` as an intermediary. The `MaxFileSize` of 10MB limits the raw file size, but after alias expansion, the in-memory representation could be much larger.

**Recommendation:** Consider setting explicit memory limits or using a YAML parser with configurable alias expansion limits. The 10MB file size cap provides some mitigation, but document this risk and consider reducing `MaxDocsPerFile` or adding a post-parse size check.

## Finding 7: No validation of suppression file fields enables silent suppression failures

**Severity: LOW**
**File:** `pkg/suppression/suppression.go`, lines 25-37
**Category:** Input Validation

`LoadSuppressions` does not validate that each suppression entry has non-empty `RuleID` and `ResourceName`. A suppression with `rule_id: ""` would never match anything (which is harmless but confusing). More problematically, a typo in a rule ID (e.g., `KC-99` instead of `KC-009`) silently does nothing, giving users a false sense that a finding is suppressed.

**Recommendation:** Validate that each suppression entry has a non-empty `rule_id` and `resource_name`. Optionally validate that `rule_id` matches a known rule. Warn on suppressions that don't match any findings.

## Finding 8: Regex-based Go template stripping is fragile and exploitable

**Severity: LOW**
**File:** `pkg/loader/loader.go`, lines 29, 157
**Category:** Input Validation / Bypass

The regex `\{\{[^}]+\}\}` used to strip Go template expressions has edge cases:
- It does not handle `}}` appearing inside string literals within the template expression.
- Nested braces like `{{ .Values.map["key}"] }}` would cause a partial match.
- The replacement with `"placeholder-value"` could create valid but misleading YAML structures that the analyzer then flags incorrectly (false positives or, worse, false negatives if the placeholder masks a dangerous pattern).

More critically, a Helm chart author could deliberately craft a template expression that survives this stripping, causing the YAML parser to fail and the file to be silently skipped (line 173: `continue`).

**Recommendation:** This is a known-hard problem (you can't fully parse templates without the template engine). The current approach is reasonable but document the limitation. Consider skipping template files entirely (detect `{{ }}` patterns and classify the file as a template rather than attempting partial parsing).

## Finding 9: Output file path is user-controlled with no sanitization

**Severity: LOW**
**File:** `cmd/kube-chainsaw/main.go`, line 122
**Category:** Path Traversal

The `--output` flag value is passed directly to `os.WriteFile`. While the tool is a CLI run by the user (so path traversal is a low-risk concern), in a CI/CD context where the output path comes from a workflow variable, an attacker who can influence the variable could write to arbitrary paths:

```go
if err := os.WriteFile(output, []byte(rendered), 0644); err != nil {
```

In the GitHub Action, `INPUT_OUTPUT` comes from `${{ inputs.output }}`, which is a workflow input. If a fork PR can influence this value, it could write to sensitive paths within the runner.

**Recommendation:** In the GitHub Action, validate that the output path is within the workspace directory. In the CLI, this is acceptable behavior for a local tool, but document it.

## Finding 10: SARIF output does not sanitize resource names/descriptions

**Severity: LOW**
**File:** `pkg/reporter/sarif.go`, lines 42-52
**Category:** Output Injection

Resource names from Kubernetes manifests are embedded directly into SARIF output as message text and location messages. A malicious manifest with crafted `metadata.name` values containing control characters, extremely long strings, or SARIF-significant JSON structures could:
- Cause truncation/corruption in SARIF viewers
- Inject misleading content into security dashboards

The `json.MarshalIndent` call handles JSON escaping, so JSON injection is not possible. However, SARIF consumers (GitHub Code Scanning, VS Code SARIF Viewer) may interpret message text in specific ways.

**Recommendation:** Sanitize resource names by restricting to the Kubernetes name regex `[a-z0-9]([-a-z0-9]*[a-z0-9])?` before embedding in output, or at minimum truncate to a reasonable length.

## Finding 11: Dockerfile uses `scratch` base image (good) but binary is not statically compiled by default

**Severity: LOW**
**File:** `Dockerfile`, `.goreleaser.yaml`
**Category:** Supply Chain / Build

The Dockerfile uses `FROM scratch`, which is ideal for minimal attack surface. The goreleaser config uses `-s -w` ldflags (strip debug info). However, CGO is not explicitly disabled (`CGO_ENABLED=0`). If the Go build picks up CGO (e.g., from the build environment), the resulting binary would dynamically link libc and fail to run in the scratch container.

**Recommendation:** Explicitly set `CGO_ENABLED=0` in the goreleaser build config:
```yaml
env:
  - CGO_ENABLED=0
```
This is likely already happening implicitly (goreleaser defaults to `CGO_ENABLED=0` for cross-compilation), but making it explicit prevents regressions.

## Finding 12: Exit code logic considers suppressed findings

**Severity: MEDIUM**
**File:** `cmd/kube-chainsaw/main.go`, lines 128-133
**Category:** Logic Correctness

The exit code loop correctly checks `!f.Suppressed` before comparing severity. This is good. However, the loop exits on the *first* matching finding via `os.Exit(1)`. If a user wants to know how many critical findings exist (e.g., from the output), the `os.Exit` prevents any post-analysis reporting or summary. It also creates a subtle ordering dependency: the exit code depends on which finding is iterated first, not a holistic assessment.

Actually, re-reading the code, the loop will call `os.Exit(1)` on the first unsuppressed finding at or above threshold. This is functionally correct for "should we fail the build?" but the `os.Exit` call (see Finding 3) is the real issue. This finding is withdrawn in favor of Finding 3.

**Status: Withdrawn (covered by Finding 3)**

## Finding 13: `escalationPodResources` is incomplete

**Severity: LOW**
**File:** `pkg/analyzer/rules.go`, lines 53-55
**Category:** False Negative / Detection Gap

The `escalationPodResources` map only includes `"pods"`. Workload controllers that create pods (deployments, daemonsets, statefulsets, jobs, cronjobs, replicasets) are not included. The KC-012 rule description says "can create pods/workloads (privilege escalation risk)" but only triggers on `pods`. A role that can create deployments (but not pods directly) would not trigger this rule, even though it's an equivalent escalation path.

**Recommendation:** Add workload controller resources to `escalationPodResources`:
```go
var escalationPodResources = map[string]bool{
    "pods":         true,
    "deployments":  true,
    "daemonsets":   true,
    "statefulsets": true,
    "jobs":         true,
    "cronjobs":     true,
    "replicasets":  true,
}
```
